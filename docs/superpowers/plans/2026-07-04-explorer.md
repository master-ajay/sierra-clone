## Insights addendum (2026-07-04)

When Insights and Explorer were merged into one product line, only the
original Explorer scope above actually got built — the Insights spec's
period-over-period trend/breakdown functionality (`docs/superpowers/specs/2026-07-04-insights-design.md`)
was never implemented. Added as an addendum here rather than a separate
service, consistent with the merge decision:

- [x] `daily_rollups` table (migration 002) — the one persisted table in an
  otherwise fully live-query product; needed because trend comparison
  requires history ADP's live queries don't retain once a window passes.
- [x] `lib/channels.ts` — fetches the channel roster (channel_id, agent_id,
  adp_user_id) from Channels' admin API, to key rollups by agent/channel.
- [x] `lib/rollups.ts` — `computeRollup` (pure, unit-tested like `metrics.ts`),
  `upsertRollup`, `getVolumeTrend`, `getAgentBreakdown`, `getChannelBreakdown`.
- [x] `POST /api/rollups/run`, `GET /api/trends/volume`, `GET /api/breakdowns/agents`,
  `GET /api/breakdowns/channels`.
- [x] `/insights` dashboard page + nav link — volume trend, percent-change,
  agent/channel breakdown tables, empty state with a rollup-now hint.
- **Scoped down from the original spec:** deflection-rate trend is not
  implemented — it needs an explicit escalation signal from Trust's audit
  log, which doesn't have one yet (same cross-product dependency the
  Insights spec's OQ-2 already flagged). Documented fast-follow, not
  silently dropped.
- **Bugs found and fixed while live-verifying this (not present in unit
  tests, which all mock the ADP client):**
  1. `lib/adp.ts`'s `fetchUserSessions` called a nonexistent
     `GET /v1/sessions?user_id=...` — ADP only exposes
     `GET /v1/users/{user_id}/sessions`. This affected the pre-existing
     `/api/metrics` endpoint too, not just the new rollup route.
  2. Channels' `create_channel` generated `adp_user_id` as a local
     `uuid.uuid4()` without ever registering that user with ADP
     (`POST /v1/users`) — so ADP had no record of it and session creation
     for a channel's synthetic user 404'd. **Fixed** (per direct request,
     commit `2c832e2` in `channels/`) — `create_channel` now registers a
     real ADP user at channel-creation time. Originally verified by
     directly patching the SQLite row's
     `adp_user_id` to a real ADP user id for this session's live test.
- 50/50 tests passing (38 original + 12 new), `next lint` + `tsc --noEmit`
  + `npm run build` all clean. Live-verified end-to-end: real ADP session/
  message data → rollup computed correctly → trend/breakdown endpoints →
  `/insights` page renders real numbers.
